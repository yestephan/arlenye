# Codebase Concerns

**Analysis Date:** 2026-03-10

## Security Considerations

**Unvalidated API Input:**
- Risk: Contact form API endpoint (`/app/api/contact/route.ts`) accepts JSON without input validation or sanitization. No checks for email format, message length, or malicious content.
- Files: `/app/api/contact/route.ts`
- Current mitigation: Client-side form validation via react-hook-form (insufficient - easily bypassed)
- Recommendations: Add server-side validation using a library like `zod` or `yup`. Validate email format, enforce max message length, and sanitize inputs before logging or processing.

**Missing Error Handling in API:**
- Risk: The contact API route has no error handling for malformed JSON or missing fields. Unhandled errors could expose stack traces or internal details.
- Files: `/app/api/contact/route.ts` (lines 4)
- Current mitigation: None - `await request.json()` will throw if JSON is invalid
- Recommendations: Wrap JSON parsing in try-catch. Return 400 Bad Request with sanitized error message for invalid input.

**Client-Side Secrets Risk:**
- Risk: Resend API key configuration is commented out but shows how secrets would be used. If uncommented without proper env var handling, could expose API keys.
- Files: `/app/api/contact/route.ts` (lines 13, 17)
- Current mitigation: Code is commented out; `RESEND_API_KEY` would be server-only but `process.env.CONTACT_EMAIL!` uses non-null assertion
- Recommendations: Document that `RESEND_API_KEY` and `CONTACT_EMAIL` must be set in `.env.local` (never in `.env`). Use non-null assertion only after explicit validation.

**Unvalidated Image Filenames:**
- Risk: Gallery component directly uses user-provided painting filenames in image paths without validation.
- Files: `/components/Gallery.tsx` (lines 25, 39)
- Current mitigation: Limited - filenames are from a static data file, not user input
- Recommendations: Ensure all filenames in `/data/paintings.ts` follow a whitelist pattern. Consider preventing arbitrary file access by validating against allowed image names.

## Testing Coverage Gaps

**No Form Validation Tests:**
- What's not tested: ContactForm validation logic, error states, submission flow
- Files: `/app/contact/ContactForm.tsx`
- Risk: Email validation edge cases, form field requirements, async submission errors could break unnoticed
- Priority: High - Contact form is user-facing and affects business

**No API Route Tests:**
- What's not tested: POST handler, JSON parsing, environment variable handling
- Files: `/app/api/contact/route.ts`
- Risk: API could fail silently or return incorrect responses without test coverage
- Priority: High - Core functionality endpoint

**No Gallery Component Tests:**
- What's not tested: Image loading, lightbox interactions, empty state rendering
- Files: `/components/Gallery.tsx`
- Risk: Gallery could break on edge cases (missing images, no paintings) without automated detection
- Priority: Medium - Critical for main feature but less likely to break

**No Integration Tests:**
- What's not tested: End-to-end contact form submission, image loading with network requests
- Risk: Component interactions could fail in production despite passing unit tests
- Priority: Medium - Would catch real-world issues

## Tech Debt

**Disabled Email Integration:**
- Issue: Email sending is completely commented out in production. Users submit forms but messages go nowhere.
- Files: `/app/api/contact/route.ts` (lines 11-21)
- Impact: Contact form collects data but fails to deliver it. Users get false success message despite no delivery.
- Fix approach: Complete the email integration by uncommenting the Resend code, adding validation, and testing with `RESEND_API_KEY` set.

**Console Logging in Production:**
- Issue: Contact form submissions logged to console via `console.log()` - this is development debugging code left in place.
- Files: `/app/api/contact/route.ts` (line 24)
- Impact: Production server logs will contain user PII (names, emails, messages). Should use structured logging instead.
- Fix approach: Replace with server-side logging library (e.g., `winston`, `pino`). Log only essential metadata, never raw user data.

**Auto-Generated Data File Without Sync Docs:**
- Issue: `/data/paintings.ts` is auto-generated by `scripts/sync-paintings.mjs` but sync process is undocumented.
- Files: `/data/paintings.ts` (line 1-2), `/scripts/sync-paintings.mjs`
- Impact: Developers may edit paintings.ts directly (changes will be overwritten). Sync script dependency unclear.
- Fix approach: Document in README how to update paintings (edit `/data/paintings-meta.json`, run `npm run sync-paintings`). Add comments warning against direct edits.

**Incomplete About Page:**
- Issue: Artist image is commented out as placeholder. Page has instructions as code comments rather than actual content.
- Files: `/app/about/page.tsx` (lines 7-15)
- Impact: About page is non-functional until image is added and code is updated.
- Fix approach: Replace commented-out Image component with actual artist photo. Remove inline instructions; update README with onboarding steps.

**Missing Environment Variable Documentation:**
- Issue: Required env vars for contact form are documented only in code comments, not in README or .env.example.
- Files: `/app/api/contact/route.ts` (lines 6-9)
- Impact: New developers won't know which env vars are needed. Missing vars cause silent failures.
- Fix approach: Create `.env.example` file listing all required variables. Update README with setup instructions.

**Loose Form Validation:**
- Issue: Form only checks for presence of fields (required: true), not email format, message length, or pattern matching.
- Files: `/app/contact/ContactForm.tsx` (lines 53, 65, 80, 93, 107)
- Impact: Invalid emails or extremely long messages could be submitted and processed.
- Fix approach: Add validation rules to react-hook-form: `pattern` for email, `minLength`/`maxLength` for message.

## Performance Bottlenecks

**Gallery Image Optimization:**
- Problem: Gallery component uses fixed width/height (600x600) for all images regardless of aspect ratio
- Files: `/components/Gallery.tsx` (lines 41-42)
- Cause: Static dimensions don't match painting aspect ratios. CSS `object-cover` crops images. Some images may be stretched or distorted.
- Improvement path: Store aspect ratio metadata in painting data (`aspectRatio` field). Use responsive sizing with CSS `aspect-ratio` property instead of fixed dimensions.

**No Image Lazy Loading:**
- Problem: All gallery images load at once on page view, even those below fold
- Files: `/components/Gallery.tsx` (lines 38-44)
- Cause: Next.js Image component defaults to eager loading. Gallery uses masonry layout that renders all at once.
- Improvement path: Add `loading="lazy"` to Image component. Consider intersecting observer for late-loading below-fold images in large galleries.

## Known Bugs

**Silent Form Submission Failures:**
- Symptoms: User clicks Submit, sees "Sending…" indefinitely. No error message appears even if API fails.
- Files: `/app/contact/ContactForm.tsx` (lines 37-39)
- Trigger: If API response is not `.ok` (status 400-599) or fetch throws, error state sets but user sees only generic message
- Workaround: Check browser console for fetch errors. Status response doesn't distinguish between server error (500) and validation failure (400).

**Mobile Navigation Doesn't Close on Navigation:**
- Symptoms: Mobile menu opens, user taps a link, menu closes due to onClick handler
- Files: `/components/Nav.tsx` (line 54)
- Trigger: Clicking a link closes menu (intentional), but navigating away and back keeps menu state closed (no re-opening on new page)
- Workaround: Menu state is component-local, not persisted. Behavior is acceptable but could be confusing if menu doesn't open on second visit.

## Fragile Areas

**Contact Form Error State:**
- Files: `/app/contact/ContactForm.tsx` (lines 15, 31-39)
- Why fragile: Error state is set when `res.ok` is false, but also caught when fetch throws. Both paths set "error" state identically without distinguishing error type. User can't know if it's network error, validation error, or server error.
- Safe modification: Add specific error messages: "Network error" for catch, "Validation error" for 400, "Server error" for 500+. Update state to include `errorMessage`.
- Test coverage: No tests for error states. Add tests for network failures, server errors, and validation failures.

**Gallery Painting Data Loading:**
- Files: `/components/Gallery.tsx` (lines 13-21), `/data/paintings.ts`
- Why fragile: If paintings array is empty, gallery shows message, but if a painting file is deleted from `/public/paintings/`, Image component fails silently. No error handling for missing images.
- Safe modification: Add error handler to Image component. Fallback to placeholder or error image if load fails.
- Test coverage: No tests for missing images. Test rendering with empty array and with missing files.

**Environment Variable Dependency:**
- Files: `/app/api/contact/route.ts` (lines 13, 17)
- Why fragile: Code assumes `process.env.RESEND_API_KEY` and `process.env.CONTACT_EMAIL` exist (non-null assertion on EMAIL). If either is missing, code will fail at runtime.
- Safe modification: Check variables exist at startup. Log warning if email sending is disabled. Return 503 Service Unavailable from API if env vars missing, not 500 error.
- Test coverage: No tests for missing env vars. Add tests for both enabled and disabled email scenarios.

## Missing Critical Features

**No Rate Limiting on Contact Form:**
- Problem: Contact API has no rate limiting. Same IP can spam unlimited requests.
- Blocks: Form cannot be safely exposed to public without spam protection
- Recommendation: Add rate limiting middleware (e.g., `@next/rate-limit` or third-party service). Limit to 5 submissions per IP per hour.

**No Input Sanitization:**
- Problem: User-submitted text (names, message) is logged as-is. Could contain HTML, scripts, or excessive whitespace.
- Blocks: Form data cannot be safely displayed in logs or UI without XSS risk
- Recommendation: Sanitize input before logging using library like `sanitize-html` or `xss`.

**No Confirmation or Delivery Feedback:**
- Problem: User has no way to know if their message was actually sent/received.
- Blocks: Reduces user confidence in contact form
- Recommendation: Send confirmation email to user. Store submissions in database for admin review.

**No Database Storage:**
- Problem: Contact submissions are only logged to console, never persisted.
- Blocks: No way to review past submissions. If server restarts, all logs are lost.
- Recommendation: Add database (Supabase, Firebase, or self-hosted). Store submissions with timestamp, IP, and status.

## Dependencies at Risk

**Resend SDK Not Yet Integrated:**
- Risk: Resend is in package.json but email functionality is disabled (commented out). Dependency usage unclear.
- Impact: If API changes, code won't be tested/found during development. Unused dependency increases bundle size.
- Migration plan: Either complete email integration (uncomment and test), or remove from package.json and dependencies if not needed.

**No Testing Framework:**
- Risk: No test runner installed (no Jest, Vitest, or other framework)
- Impact: Cannot write automated tests. All testing is manual.
- Migration plan: Add `vitest` or `jest` to devDependencies. Set up test configuration. Start with contact form tests (highest priority).

## Scaling Limits

**Static Painting Data:**
- Current capacity: Hardcoded array in memory. Manual sync script required for updates.
- Limit: Only 2 paintings currently. Adding 100+ would require database instead of static file.
- Scaling path: Migrate to database (Supabase/Firebase). Move painting metadata to database. Implement gallery pagination or lazy loading for large collections.
